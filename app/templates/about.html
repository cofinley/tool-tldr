About

TL;DR: it's a wiki for any kind of software tool. Good luck and have fun.

Tool TL;DR was created to help people of all levels of expertise get a grasp of the growing software tool landscape. Whether you're a beginner who doesn't know what's out there, or a senior dev who just wants a quick rundown of a particular framework/library/tool, you will find answers here. The explanations are not comprehensive by any means (hence the name), but they provide a quick, clear, and helpful description without having to dig through the docs. And, if you feel like a particular description could be improved, feel free to edit! Tool TL;DR is, and will always be, a community-first place.

Organization

The wiki is broken down by categories and tools. The tools fit into the appropriate categories (i.e. Eclipse would fall under the IDE category). Categories can also be subcategories of other categories to improve organization. For instance, jQuery might fall under the Front-end category which might go in the Web Development category. The organization is determined by the community: you can rearrange the structure to find a better fit for a tool or category if it needs a more suitable home.

Format

Categories

Categories have three main parts: the what, why, and where.

What

The what is pretty self-explanatory. It should provide the TL;DR of the category and explain what the category's tools do.

Why

The why asks why the category of tools is needed in the first place. For those just starting out, this can be valuable when you don't know what you don't know. It also aims to answer why someone would need to start using the type of tool. Again, for those exploring the landscape for the first time, this can provide a great explanation to a question that they didn't even know they might have. For example, caching is invaluable for a web application, but a beginner might not know that they need it for reducing redundant trips to get data or images, thereby saving money and improving site load time by reducing bandwidth.

Where

The where is a little different. It asks where in an application the type of tool would go. For example, the Web Server category would describe tools that go in front of an Application Server and would be the first in line to handle incoming web traffic. In another case, the MVC Framework category would house tools such as AngularJS, and would be found in the front-end as HTML and Javascript, running in the browser and not the server. The where question exists simply to help people create a mental image of all the pieces in the puzzle.

Tools

Tools have one main question: why use this tool over one of its alternatives in the same category?

Why

Tools will often have neighbors in the same space, but when exploring what's out there in a TL;DR fashion, you probably want answers as to what separates the tools quickly and succinctly. This section, just like the others, cuts out the jargon and elucidates any blurred lines.

Tool Details

There are several other descriptors for tools on their pages.

  • Environment: the type of programming environment this tool is found. This is usually a programming language.
  • Created: the year this tool was created or launched.
  • Version: the latest version of the tool.
  • Active: the status of the tool (yes or no).
  • Project URL: the link to the home of the project.
